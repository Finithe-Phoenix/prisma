## Key mapping decisions (verified by independent derivation)

- The x86↔ARM semantics were derived twice independently (one agent
  with no repo access, plus the implementing session) and agreed on
  6/7 instructions; the derivation **improved** the seventh:
  SHA256RNDS2 IS realizable on ARM's 4-round SHA256H/SHA256H2 with
  WK2=WK3=0 because the shift-register recurrence keeps the garbage
  rounds out of output elements 2..3 (they depend only on WK0/WK1).
- SHA1SU0 is deliberately NOT used for SHA1MSG1: it folds the
  W[t-8] XOR term that x86 defers to the caller's PXOR. A negative
  lowering assert pins this.
- Emission is unconditional (the AES/Aese precedent): vixl at the
  pinned commit provides all ten crypto encodings and every target
  host (Apple silicon, Graviton/Cobalt CI runners) has
  FEAT_SHA1/SHA256. HostFeatures detection + guest CPUID
  advertisement remain queued (CPUID currently zeroes everything,
  so well-behaved guests will not take SHA-NI paths yet).

### Gemini triage

| # | Sev claimed | Verdict | Why |
|---|---|---|---|
| 1 | BLOCKER | False positive | Claims SHA1NEXTE should rotate the SOURCE. The SDM rotates SRC1 — which IS the destination register (xmm1) — and copies lanes 95:0 from SRC2. The canonical Intel loop (`sha1nexte E0, MSG` → MSG + {E,0,0,0}) is only coherent with the implemented direction. |
| 2 | MAJOR | Falls with #1 | The host reference matches the SDM, not "the same bug". The references transcribe x86 semantics only, so they cannot share an ARM-side lowering bug. |
| 3 | MAJOR | False | Mis-recalls ARM crypto lane order as A-in-element-3. The ARM ARM pseudocode reads `SHAchoose(Y<31:0>, …)` — E/A live in element 0 — and the 256-bit ROL pushes each new A into X<31:0>, so after 4 rounds X = {A4,A3,A2,A1} and {A2,A1} sit in the HIGH half (Zip2 correct). The independent derivation reached the same layout; any real flip would fail the SHA256RNDS2 e2e on the ARM64 CI runner. |
| 4 | NIT | Confirmed correct | Reviewer itself verifies the vixl Ext operand order (Vn = low half). |
| 5 | NIT | Confirmed correct | Reviewer itself verifies the SHA256MSG2 lane-0 clear. |
