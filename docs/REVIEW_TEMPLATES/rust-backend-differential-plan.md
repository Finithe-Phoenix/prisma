# Rust Backend Differential Plan

Scope: plan the first byte-level differential harness between the C++ ARM64
backend and the Rust `prisma-backend` migration slice. This is intentionally
limited to scalar IR already covered by both sides:

- `Constant`
- `BinOp Add/Sub` where the RHS is a 12-bit immediate constant
- `BinOp And/Or/Xor`
- `BinOp Shl/Shr/Sar/Ror`
- `CmpFlags` followed by `CondJumpFlags`
- `Jump`, boolean `CondJump`, and `JumpReg`
- sized `LoadMem` / `StoreMem` for `I8`, `I16`, `I32`, and `I64`

Do not expand this harness to SIMD, TSO memory, calls, relative guest exits,
spilling, or broad decoder-driven input until this scalar slice is stable.

## Existing APIs

C++ lowering path:

```cpp
backend::Emitter em;
backend::Lowerer lw(em);
const auto res = lw.lower(fn); // or lw.lower(std::span<const ir::Stmt>)
REQUIRE(res.success);
em.finalize();
const auto bytes = em.code_bytes();
```

Rust lowering path:

```rust
let words = prisma_backend::Lowerer::new().lower_function(&func)?;
let bytes = words
    .iter()
    .flat_map(|word| word.to_le_bytes())
    .collect::<Vec<u8>>();
```

`Arm64Assembler::finish_bytes()` is available for assembler-level tests, but
`Lowerer::lower_function()` currently returns `Vec<u32>`, so the first
differential harness should normalize Rust words to little-endian bytes in the
test helper.

## Minimal Harness Shape

Add a single focused C++ test file later, for example
`core/tests/test_rust_backend_differential.cpp`, only after choosing the
cross-language bridge. The helper should:

1. Build named `ir::Function` fixtures in C++.
2. Lower each fixture through `backend::Lowerer`.
3. Lower the equivalent serialized fixture through Rust `prisma_backend`.
4. Compare finalized ARM64 bytes exactly.
5. On mismatch, print fixture name, C++ bytes, Rust bytes, and C++ disassembly.

Prefer a tiny fixture registry over generated random IR for the first pass.
The goal is to pin known migration coverage, not to fuzz unsupported surfaces.

## Corpus

Start with one fixture per backend behavior, then add sizes/conditions as table
rows:

- `const_i64_store_ret`: constant materialization plus `Return`.
- `add_imm12` / `sub_imm12`: lhs register value plus RHS constants `0`, `1`,
  `4095`; reject or skip `4096`.
- `logic_i64`: `And`, `Or`, `Xor` over two constants/register-backed refs.
- `shift_i64`: `Shl`, `Shr`, `Sar`, `Ror` with counts `0`, `1`, `31`, `63`.
- `cmpflags_condjumpflags_widths`: `CmpFlags` on `I8`, `I16`, `I32`, `I64`,
  then `CondJumpFlags` for `Eq`, `Ne`, `Ult`, `Slt`.
- `jump_forward_backward`: three-block function with direct forward and
  backward `Jump`.
- `condjump_bool`: `CondJump` true/false edge pair from a constant condition.
- `jump_reg`: constant target ref lowered through `JumpReg`.
- `load_store_widths`: `StoreMem` then `LoadMem` for `I8`, `I16`, `I32`,
  `I64`, using an address constant that is already materialized in a register.

## Exact First Command/API

After the sidecar test target exists, the first differential command should be
targeted, not broad:

```powershell
cmake --build core/build --target prisma_core_tests
core/build/prisma_core_tests --reporter compact "RustBackendDifferential: scalar migration slice"
cargo +1.95.0-x86_64-pc-windows-msvc test -p prisma-backend
```

The first API assertion should be this single fixture:

```cpp
CHECK(rust_backend_bytes("const_i64_store_ret", fn) == cpp_lower_bytes(fn));
```

where `cpp_lower_bytes(fn)` is exactly:

```cpp
std::vector<std::uint8_t> cpp_lower_bytes(const ir::Function& fn) {
    backend::Emitter em;
    backend::Lowerer lw(em);
    const auto res = lw.lower(fn);
    REQUIRE(res.success);
    em.finalize();
    const auto bytes = em.code_bytes();
    return {bytes.begin(), bytes.end()};
}
```

## Blockers

- There is no checked-in C++ to Rust differential bridge yet. The harness needs
  either a small C ABI exported by `prisma-backend`, a test-only subprocess
  that consumes serialized `prisma-ir`, or a generated byte oracle committed by
  the Rust test. Prefer the C ABI or subprocess; avoid committed oracles for
  label-heavy branch tests.
- C++ and Rust IR types are parallel definitions, not one shared in-process
  type. The bridge must choose one serialization format before byte comparison
  can be automated.
- Rust `Lowerer::lower_function()` returns words, while C++ `Emitter` returns
  bytes. Normalize to little-endian bytes before comparing.
- Label fixups can differ if block order differs. Use the same explicit
  `Function.blocks` order on both sides and keep entry block first in the
  initial corpus.
- Keep unsupported ops out of the corpus. In particular, do not mix in
  `LoadMemTSO`, `StoreMemTSO`, `JumpRel`, `CondJumpRel`, `Call*`, `RetAdjusted`,
  SIMD, FP, or spill-pressure cases.
